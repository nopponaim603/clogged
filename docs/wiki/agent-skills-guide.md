## 1. Caveman
**วัตถุประสงค์:** โหมดลดการใช้ Token และประหยัดพลังงานการสนทนาได้มากถึง ~75% โดยลดการพูดคุยที่เยิ่นเย้อ แต่ยังคงความถูกต้องของข้อมูลเชิงเทคนิคและคำสั่งโค้ดไว้อย่างครบถ้วน

### 📌 วิธีการกระตุ้น (Trigger)
คุณสามารถสั่งให้เปิดโหมดนี้ได้โดยการพิมพ์คำหรือประโยคดังต่อไปนี้ในแชท:
* `/caveman`
* `"caveman mode"`
* `"talk like caveman"`
* `"use caveman"`
* `"be brief"` หรือ `"less tokens"`
* หากต้องการปิด ให้พิมพ์: `"stop caveman"` หรือ `"normal mode"`

### ⚙️ ระดับความเข้มข้น (Intensity Levels)
คุณสามารถเปลี่ยนระดับได้ผ่านคำสั่ง `/caveman [ระดับ]` เช่น `/caveman ultra` หรือ `/caveman lite`

| ระดับ | รายละเอียดการทำงาน | ตัวอย่างผลลัพธ์ (คำถาม: Why re-render?) |
|---|---|---|
| **lite** | ตัดประโยคอรัมภบทและคำฟุ่มเฟือย แต่คงความถูกต้องของไวยากรณ์ | "Your component re-renders because you create a new object reference each render. Wrap it in `useMemo`." |
| **full** (Default) | ตัดบทความ (a/an/the) ใช้ประโยคสั้น/วลี ตัดการอธิบายเครื่องมือ | "New object ref each render. Inline object prop = new ref = re-render. Wrap in `useMemo`." |
| **ultra** | ย่นย่อคำทั่วไปในส่วนคำอธิบาย (เช่น auth, config, req, res) แต่**ห้าม**ย่อโค้ดจริง | "Inline obj prop → new ref → re-render. `useMemo`." |
| **wenyan-lite** | ปรับสไตล์คำตอบเป็นภาษาจีนโบราณแบบกระชับ (กึ่งคลาสสิก) | "組件頻重繪，以每繪新生對象參照故。以 useMemo 包之。" |
| **wenyan-full** | บีบอัดภาษาจีนโบราณเต็มรูปแบบ (文言文) ลดตัวอักษร 80-90% | "每繪新生對象參照，故重繪；以 useMemo 包之則免。" |
| **wenyan-ultra** | บีบอัดภาษาจีนโบราณขั้นสุด | "新參照→重繪。useMemo Wrap。" |

## 2. Context Canary
**วัตถุประสงค์:** ทำหน้าที่เป็น "นกคานารีในเหมืองถ่านหิน" เพื่อตรวจจับอาการหลงลืม (Context Rot / Drift) ของ AI ในระหว่างการแชทระยะยาวแบบเรียลไทม์

### 📌 วิธีการกระตุ้น (Trigger)
* พิมพ์คำสั่ง: `"canary"`, `"context canary"`, `"canary check"`
* หรือทักเมื่อพบว่า AI ลืมตัว: `"you stopped using my name"` หรือ `"did you lose context"`
* หรือถามตรงๆ: `"how degraded is your context"`

### ⚙️ หลักการทำงาน (Canary Contract)
เมื่อเปิดใช้งาน AI จะต้องแสดงสัญลักษณ์ Canary ไว้ที่**บรรทัดแรกสุดของทุกการตอบกลับ** เสมอ เช่น:
```markdown
**Noppon · t14 · ctx ok**
```
* **Noppon (ชื่อคุณ):** หากชื่อนี้หายไป แปลว่าคำสั่งระบบหลุดจากหัวข้อการประมวลผลหลักแล้ว
* **t14 (Turn Counter):** ตัวนับจำนวนรอบของการตอบกลับ (หากข้ามหลักหรือเลขผิดปกติ แสดงว่าเกิดการ Compaction/บีบอัดประวัติแชท)
* **ctx ok (สถานะ Context):** AI จะสแกนตัวเองและประเมินว่ามีอาการหลงลืมหรือไม่ (`ctx ok` = ปกติ, `ctx aging` = เริ่มลืมรายละเอียด, `ctx thin` = ลืมการตัดสินใจรอบแรกๆ ไปแล้ว)

### 🚨 เมื่อตรวจพบว่านกคานารีตาย (Canary Trips Protocol)
หากสัญลักษณ์นี้หายไป 2 รอบการคุยติดต่อกัน หรือตัวเลขผิดปกติ ให้ทำตามขั้นตอนการกู้คืน:
1. **หยุดทำงานทันที** อย่าเพิ่งฝืนเขียนโค้ดต่อด้วยระบบที่กำลังหลงลืม
2. **สร้าง Checkpoint:** สั่งให้เขียนไฟล์สถานะปัจจุบันลงโฟลเดอร์โปรเจกต์ เช่น `HANDOFF.md` เพื่อบันทึกสิ่งที่ทำเสร็จแล้วและสิ่งที่ต้องทำต่อ
3. **เริ่มต้นแชทใหม่ (Reset Deliberately):** นำเนื้อหาจากไฟล์สรุปไปเริ่มสนทนาในหน้าต่างใหม่ (Session ใหม่) เพื่อล้างหน่วยความจำให้สะอาดและป้องกันข้อผิดพลาดเชิงลึก

## 3. Fuck Slop
**วัตถุประสงค์:** ล้างโครงสร้างและพฤติกรรมงานเขียนที่ดูเป็น "AI เกินไป" (เช่น การใช้รูปประโยคยอดฮิต "not just X, but Y", การใช้ Em-dash บ่อยๆ, ลิสต์สามอย่าง (Rule-of-three), และคำอวยเนื้อหาที่เว่อร์เกินจริง) เพื่อเปลี่ยนให้เป็นงานเขียนแบบที่มนุษย์เขียนจริง

### 📌 วิธีการกระตุ้น (Trigger)
* ส่งงานเขียนให้ขัดเกลาพร้อมบอกว่า: `"fuck slop"`, `"deslop this"`, `"remove AI tells"`, `"humanize this"`
* หรือพิมพ์สั่งโดยตรงผ่าน: `/fuck-slop`

### ⚙️ ขั้นตอนการทำงานของ Skill
1. **รับเนื้อหาและวิเคราะห์สไตล์ / Genre**
2. **สแกนเชิงกลเพื่อหา AI tells** (เช่น regex ลิสต์สามประการ, คำฟุ่มเฟือย)
3. **เขียนใหม่ตามเจตจำนง** - ตัดประโยคอวย/ประโยคฟุ่มเฟือย
4. **สแกนความสะอาดซ้ำ Verify Loop** จนไม่พบ AI tells
5. **ส่งมอบเนื้อหาพร้อมรายละเอียดสรุป**ส่วนที่เปลี่ยน

### 💡 ตัวอย่างการเปลี่ยนประโยคให้ดูเป็นมนุษย์
* **AI Slop:** *"It's not just a tool, it's a fundamental shift in how teams work"* (มันไม่ใช่แค่เครื่องมือ แต่คือการปฏิวัติวิธีการทำงาน...)
* **Human-written:** *"Teams that adopted it stopped holding standups within a month."* (ทีมที่นำมันไปใช้สามารถเลิกประชุม Standup ได้ภายในเวลาไม่ถึงเดือน)

## 4. Grill Me
**วัตถุประสงค์:** เปิดโหมด "ซักฟอกแผนงาน" เพื่อทดสอบและวิเคราะห์สมมติฐาน ความเสี่ยง ตลอดจนจุดเปราะบางในแผนของคุณก่อนจะเริ่มลงมือเขียนโค้ดจริง

### 📌 วิธีการกระตุ้น (Trigger)
* พิมพ์ประโยค: `"grill me"`, `"stress-test this"`, `"challenge my plan"`, `"interview me"`
* หรือพิมพ์คำสั่ง: `/grill-me`

### ⚙️ รูปแบบการสนทนา
1. **Calibration (ปรับระดับ):** AI จะถามระดับความรู้และความแรงที่คุณต้องการ:
   * *ระดับความรู้:* New (ผู้เริ่มต้น) / Working (พอทำได้) / Expert (ผู้เชี่ยวชาญ)
   * *ระดับแรงกดดัน:* Light (เน้นเคลียร์สเปก) / Standard (ท้าทายทางเลือก) / Hard (เจาะหาจุดพังใน Production)
2. **One Question at a time:** AI จะตั้งคำถาม**ทีละข้อเท่านั้น** และจะมี *Recommended Answer (ตัวเลือกแนวคำตอบ)* พร้อมเหตุผลประกอบเสมอเพื่อให้คุยง่ายขึ้น เช่น:
   > **Question:** What outcome matters most?
   > **Recommended answer:** "We must support 500 requests per second under 200ms latency."
   > **Why it matters:** Setting concrete boundaries avoids scope creep.

## 5. Interface Kit
**วัตถุประสงค์:** คู่มือระดับวิศวกรเพื่อสร้างหน้าจอผู้ใช้งาน (UI) ที่มีสุนทรียภาพ สวยงามพรีเมียม (Rich Aesthetics), เข้าถึงได้ง่าย (Accessibility/a11y), และมีไมโครอนิเมชันที่ลื่นไหลตามหลักการพัฒนาเว็บสมัยใหม่

### 📌 วิธีการกระตุ้น (Trigger)
* พิมพ์สั่งให้สร้าง UI หรือปรับแต่งดีไซน์: `"build UI"`, `"create component"`, `"make it look good"`, `"polish UI"`, `"make it beautiful"`
* หรือคำที่เกี่ยวข้องกับ Frontend เช่น `"landing page"`, `"CSS styling"`, `"animations"`

### ⚙️ สิ่งที่คุณจะได้รับจาก Skill นี้
* **Rich Color Palette:** การใช้ HSL/CSS variables ปรับเปลี่ยนเฉดสีเป็นธีม Dark Mode ที่กลมกลืน แทนที่จะใช้สีพื้นฐานทั่วไป
* **Micro-animations:** การเพิ่ม Transition เมื่อนำเมาส์ไปชี้หรือกดปุ่ม เพื่อให้หน้าจอดูมีมิติและน่าใช้งาน
* **Accessibility (a11y):** ควบคุม Contrast ratio ของสีตัวอักษร, การทำ focus ring, การรองรับ Screen Reader

## 6. Junior to Senior
**วัตถุประสงค์:** ทำหน้าที่รีวิว "แผนงาน/แผนสถาปัตยกรรมระบบ" เสมือนมี Senior Developer/Staff Engineer มาช่วยแกะรอย ค้นหาช่องโหว่ประเภทแผนงานคลุมเครือ (Fog) หรือแผนงานอุดอู้เน้นจุดเล็กแต่ลืมภาพรวม (Tunnel)

### 📌 วิธีการกระตุ้น (Trigger)
* สั่งเมื่อมีแผนหรือเขียนแผนเสร็จ: `"junior to senior"`, `"senior review"`, `"review this like a staff engineer"`

### ⚙️ การประเมินแผนงานโดย Senior
* **Fog Test (สแกนความพร่ามัว):** ดูว่าหัวข้อต่างๆ เช่น "ทำการ Cache ข้อมูล" เขียนขึ้นมาลอยๆ หรือไม่? Senior จะเรียกร้องให้ระบุตัวแปร โครงสร้างข้อมูล และไลบรารีที่จะใช้จริง
* **Tunnel Test (สแกนการมองมุมแคบ):** ดูว่าแผนเจาะลึกเฉพาะจุดเกินไปจนลืมระบุว่าฟีเจอร์นี้ทำไปเพื่อใคร ข้อดีที่ชนะทางเลือกเดิมคืออะไร และไม่มีขอบเขตงาน (Non-goals) ชัดเจนหรือไม่
* **SOTA & Lockfile Check:** ตรวจสอบประวัติโค้ดจริงและเวอร์ชันไลบรารีในโปรเจกต์ เพื่อไม่ให้เสนอวิธีที่ล้าสมัยหรือทำงานร่วมกับตัวเก่าไม่ได้

## 7. Last 20%
**วัตถุประสงค์:** โฟกัสไปที่ 20% สุดท้ายของการส่งมอบงาน (Experiential Layer) ที่ AI มักข้ามไป เช่น การทำหน้าจอสถานะไม่มีข้อมูล (Empty State), หน้าจอดาวน์โหลดครั้งแรก (First-run State), และทางลัดการใช้งานที่สมบูรณ์ (Golden Path) เพื่อไม่ให้ระบบดูเป็นแค่ตัวเดโมที่ใช้งานจริงไม่ได้

### 📌 วิธีการกระตุ้น (Trigger)
* บอกตัวแทนเมื่อสร้างฟีเจอร์ใกล้เสร็จ: `"last 20%"`, `"finish this"`, `"make it actually good"`, `"is this actually done"`, `"why does this feel flat"`

### ⚙️ วิธีเช็กคุณภาพ
* **Golden Artifact:** ร่างหน้าเอกสารตัวอย่างที่ดีที่สุด (เช่น หน้า Wiki หรือรายงานผลที่มนุษย์เขียนจัดเต็ม) ขึ้นมาก่อนจะเริ่มสร้างระบบสุ่มเจเนอเรต เพื่อเป็นเกณฑ์วัดว่า AI ต้องสร้างออกมาได้สวยงามทัดเทียม
* **The Walk (การทดลองเดินแผน):** แกล้งเป็นผู้ใช้งานทั่วไปที่ไม่มีข้อมูลเชิงลึกในระบบ เข้าไปกดเล่น ลองคลิก ลองค้นหา และจับตาดูว่าเกิดช่องโหว่ความน่ารำคาญตรงไหนหรือไม่ ก่อนจะส่งมอบงานจริง

## 8. Loop Factory
**วัตถุประสงค์:** รันระบบท่อส่งงานพัฒนาอัตโนมัติแบบกำหนดสเปก (Spec-driven Development) โดยทำงานตามโฟลเดอร์ของ Spec-file ที่ย้ายไปมาเพื่อควบคุมความโปร่งใส

```text
    inbox/           active/           archive/
 (งานรอพัฒนา)  -->  (กำลังโค้ด/ทดสอบ) -->  (ส่งงาน/เก็บเข้ากรุ)
```

### 📌 วิธีการกระตุ้น (Trigger)
* พิมพ์สั่งการรันงานระบบลูป: `/loop-factory` หรือพิมพ์คำสำคัญ `"agent factory"`, `"loop factory"`, `"spec-driven loop"`

### ⚙️ คำสั่งหลักบนระบบ Terminal
หากโปรเจกต์ของคุณทำการติดตั้ง Loop Factory CLI เรียบร้อยแล้ว (โดยมีโครงสร้างโฟลเดอร์ `factory/specs/`) คุณสามารถรันคำสั่งเหล่านี้ได้:

* `loop-factory scan`: สแกนดูไฟล์งานทั้งหมดใน Inbox และ Active
* `loop-factory dispatch --agent claude --stage`: เตรียม prompt สำหรับส่งโค้ดเข้าสู่ช่วง Active
* `loop-factory review [spec-id] --agent claude`: เรียกคำสั่งให้รีวิวเปรียบเทียบ Code diff กับ Acceptance Criteria ในตัว Spec
* `loop-factory archive [spec-id] --accepted`: จัดเก็บงานที่ผ่านการอนุมัติแล้วอย่างเป็นทางการ
